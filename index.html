Governance Protocol
// Rules for how the system evolves · ADD · MODIFY · REMOVE · DEPRECATE
PRINCIPLE 01
No Agent Without Registration
Every agent that touches any EPIK pipeline must have a row in the Agent Registry before it processes a single task. No exceptions. Unregistered agents are invisible to QA gates, the Orchestrator, and the changelog.
PRINCIPLE 02
No Rubric, No Gate
Every QA gate must reference a named rubric file in the Memory Layer. A gate without a rubric file is not a gate — it's a placeholder that will pass or fail randomly. Write the rubric before you build the agent.
PRINCIPLE 03
Dependency Check Before Any Change
Before modifying, removing, or replacing any agent, run a dependency check in the Registry. Shared agents (PRD, UI/UX, Coder) affect multiple pipelines. A change to one without checking dependents will break downstream silently.
Protocol Definitions
➕
ADD Protocol
Registering a new agent or pipeline
▼
1
Define the agent
Write a 1-paragraph description of what the agent does, what it does NOT do, and which pipeline it belongs to.
2
Write the rubric file first
If this agent produces an output that passes through a QA gate, write the rubric file before writing the system prompt. Store it in Memory Layer under qa_rubrics/.
rubric file named and stored in Notion
minimum 5 checklist items
gate type defined: AUTO / HUMAN / ASYNC
3
Register in Agent Registry
Add a row to the Agent Registry Notion DB with all required fields before the agent processes any task.
agent_id assigned (format: PIPE-NAME, e.g. DEV-PRD)
status set to BUILDING
owner named
dependencies listed
rubric_file linked
4
Update EPIK Brain routing
If this is a new Domain Lead or a new pipeline entry point, update EPIK Brain's system prompt to recognize the new routing destination.
🛡️ GATE — Orchestrator must be updated BEFORE the agent goes live or tasks will be lost.
5
Wire n8n trigger
Add the routing webhook or trigger in n8n. Test with a dry-run task before marking status ACTIVE.
6
Update Registry status → ACTIVE + log to Changelog
Once dry-run passes, update Registry row to ACTIVE. Add a changelog entry in Tab 04 with type ADD.
✏️
MODIFY Protocol
Changing an existing agent's behavior or tools
▼
1
Check dependencies
Look up the agent in Registry. Check the "Depends On" and "Used By" fields. If the agent is SHARED (e.g. PRD Agent), list every pipeline that will be affected.
🛡️ GATE — If agent is shared across 2+ pipelines, notify all pipeline owners before making changes.
2
Assess change scope
Minor change (system prompt wording, tool swap): proceed directly. Major change (new QA gate, new output format, pipeline restructure): requires human sign-off from Akhri.
3
Update rubric file if output changes
If the agent's output format changes, update the rubric file to match. Increment the rubric version number. Old rubric version is archived, not deleted.
4
Test in isolation before live
Run 3 test tasks through the modified agent in a staging context before it handles real tasks.
5
Update Registry + Changelog
Increment version in Registry row. Add Changelog entry with type MODIFY, listing what changed and why.
🗑️
REMOVE Protocol
Permanently removing an agent from the system
▼
1
Dependency audit
Check who depends on this agent. If any other agent lists it as a dependency, those agents must be updated or removed first. No orphaned dependencies allowed.
🛡️ GATE — Cannot remove if active dependencies exist. Resolve dependencies first.
2
Drain active tasks
Check task log for any in-flight tasks assigned to this agent. Complete or manually redirect all active tasks before removal.
3
Remove routing from n8n and EPIK Brain
Delete the trigger in n8n. Update EPIK Brain's system prompt to remove routing references. Test that EPIK Brain handles requests that would have gone to this agent gracefully.
4
Mark DEPRECATED in Registry
Never delete a Registry row — set status to DEPRECATED with a deprecation date and reason. Archived rubric files are kept. System prompt is archived, not deleted.
5
Log to Changelog
Add Changelog entry with type REMOVE, including the reason for removal and what (if anything) replaced it.
🔀
NEW PIPELINE Protocol
Adding an entirely new pipeline equivalent to Media or Dev Studio
▼
1
Define pipeline scope and ownership
A pipeline needs: a named domain, a human owner, a clear entry point (what triggers it), and a clear exit point (what it delivers). Write this 1-pager before anything else.
2
Identify shared vs new agents
Check the Registry. Can existing agents (PRD, Coder, etc.) be reused with context mode? If yes, use MODIFY Protocol on them. If no, use ADD Protocol for each new agent.
🛡️ GATE — Default to reusing shared agents. Only create new agents when the function is genuinely different.
3
Write all rubric files for new QA gates
Every QA gate in the new pipeline needs a rubric file before any agent is built. Minimum 5 checklist items per rubric.
4
Add Domain Lead to EPIK Brain routing
The new pipeline's Domain Lead must be registered in the Agent Registry and added to EPIK Brain's routing logic before any task can enter the pipeline.
5
Add Memory Layer section
Create a new folder in the Memory Layer Notion DB for this pipeline's context documents. At minimum: product/domain spec + brand/voice guide if applicable.
6
Wire n8n + run dry-run end-to-end
Build the full trigger chain in n8n. Run 3 dry-run tasks through the complete pipeline before going live. All QA gates must fire correctly in dry-run.
7
Log all agents + pipeline to Registry and Changelog
Every agent in the new pipeline gets a Registry row. One Changelog entry per agent (type ADD) + one entry for the pipeline itself.
